# Implementation Plan: Community Hub UI Redesign & Missing API Mocking

**Branch**: `002-ui-redesign-api-mocking` | **Date**: 2026-05-29 | **Spec**: [spec.md](file:///d:/Microservices-Online-Judge-System/specs/002-ui-redesign-api-mocking/spec.md)

**Input**: Feature specification from `specs/002-ui-redesign-api-mocking/spec.md`

## Summary

The JudgeSync Community Hub frontend (`src/Web`) is a React 19 SPA built in a single-file architecture (`App.jsx`). It currently implements articles, comments, votes, bookmarks, and recommendations using a `useApi()` hook, a `useResource()` secondary-data hook, and a `PageErrorBoundary`. This plan redesigns all 21 screens using the Terminal Nexus Tailwind-based design system, expands the routing table with 15 new routes, implements 14 missing backend features end-to-end in the React state machine, mocks all missing endpoints with MSW v2, and compiles full API contract documentation.

## Technical Context

**Language/Version**: JavaScript (ES Modules) — React 19.2.4

**Primary Dependencies**: React 19, react-router-dom 7, lucide-react 1.7, `@tailwindcss/vite` (new), `msw` v2 (new), `@monaco-editor/react` 4.7 (existing)

**Storage**: `localStorage` (`communityToken` key for JWT)

**Testing**: MSW v2-backed manual visual testing; all mocks simulate loading, empty, and error states

**Target Platform**: Chrome, Firefox, Safari, Edge — responsive from 375px to 1440px

**Project Type**: Single Page Application — `src/Web/` (Vite 8 + React 19)

**Performance Goals**: Page-critical data renders within 1s on broadband; search suggestions within 300ms; loading states visible for ≥100ms

**Constraints**: 
- Single-file architecture (`App.jsx`) — all page components live here; shared utilities and hooks defined at the top
- Tailwind CSS authorized for all new/redesigned components (per Constitution XIII)
- No additional npm dependencies beyond Tailwind CSS and MSW v2
- MSW worker only active in development mode (`import.meta.env.DEV`)

**Scale/Scope**: 21 screens; 15 new routes; 29 functional requirements; ~14 new MSW mock handlers

## Implementation Phases

### Phase 0: Infrastructure Setup

1. **Install new dependencies**:
   - `@tailwindcss/vite` as devDependency
   - `msw@2` as devDependency
2. **Configure Tailwind in `vite.config.js`**: Add the `@tailwindcss/vite` plugin.
3. **Create a CSS entry point** for Tailwind: Add `@import "tailwindcss"` to `index.css` (or a new `tailwind.css`) and import it in `main.jsx`.
4. **Initialize MSW**: Run `npx msw init public/ --save` to generate `public/mockServiceWorker.js`.
5. **Create `src/mocks/browser.js`**: Set up and export the MSW service worker.
6. **Create `src/mocks/handlers.js`**: Scaffold file with empty handlers array (populated phase-by-phase).
7. **Update `src/main.jsx`**: Conditionally start MSW worker in `DEV` mode before rendering the React tree.

### Phase 1: Foundation Hooks & Auth Flow

Add the following shared hooks at the top of `App.jsx` (before all component definitions):

1. **`useAuth(api)`** — Wraps token expiry detection:
   - Patches `api.request` to catch HTTP 401 responses
   - On 401: clears token via `api.setToken("")` and calls `navigate("/auth?returnUrl=" + encodeURIComponent(location.pathname + location.search))`
   - Used by all authenticated page components

2. **`useNotificationCount(api)`** — Navigation-driven notification poll:
   - Uses `useLocation()` to detect URL changes (path, query, hash)
   - On each location change, fetches `GET /api/notifications/unread-count` if authenticated
   - Returns `{ count, loading }`
   - MSW handler added in `handlers.js`

3. **`usePresignedUpload()`** — Avatar upload helper:
   - Accepts `uploadUrlEndpoint` and `file`
   - Fetches presigned URL from backend, then PUTs the file directly to the storage provider URL
   - Returns `{ upload, uploading, error, progress }`

### Phase 2: Global Shell & Navigation Redesign

Redesign the `Shell` and `Header` components using Tailwind classes to match the Terminal Nexus design system:

1. **Landing Page** (`WelcomePage` component):
   - Public route `/welcome`; redirect unauthenticated root visitors here
   - Static page: hero section, feature cards, CTA buttons linking to `/auth`
   - No API calls

2. **Auth Page** (`AuthPage` component):
   - Route `/auth`; reads `?returnUrl` from query string
   - Tabbed login / register forms with client-side validation and error banners
   - On success: stores token via `api.setToken()` → navigates to `returnUrl` or `/`
   - Wired to real backend: `POST /users/login`, `POST /users/register`

3. **Redesigned `Header`**:
   - Notification bell with unread count badge (via `useNotificationCount`)
   - Global search trigger button (opens `SearchOverlay`)
   - Auth-aware: show login link when unauthenticated; avatar + profile menu when authenticated

4. **`SearchOverlay` component** (global, mounted in `Shell`):
   - Cmd+K palette-style overlay with backdrop blur
   - Debounced input via existing `useDebouncedValue` hook
   - Fetches `GET /api/search/suggestions?q=...` — **MSW mocked**
   - Results grouped by Articles, Tags, Authors

5. **Update `Shell` routing table** — add all new routes:

### Phase 3: Core Feed & Article Screens (Real API)

Redesign existing components with Tailwind — all backed by real Community API endpoints:

1. **`Dashboard`** (home feed `/`) — Redesign card layout, sort controls, tag filter pills, sidebar metrics.
2. **`ArticleDetails`** (`/articles/:slug`) — Guest read-only; interactions prompt auth. Single-level reply thread with vertical guide lines. Upvote/downvote with optimistic update + rollback on error.
3. **`ArticleEditor`** (`/articles/new`, `/articles/:slug/edit`) — Monaco editor in body field; tag input; cover image via presigned URL (`POST /api/articles/{id}/cover-image-upload-url`).
4. **`ArticleCollection`** (`/me`, `/bookmarks`) — Tabbed view: My Articles / Bookmarks.

### Phase 4: Discovery & Search Screens (MSW Mocked)

Implement full state machines (loading, data, empty, error) for:

1. **`ExplorePage`** — Curated trending content grid. MSW: `GET /api/explore/highlights`.
2. **`TagsExplorerPage`** — Tag directory grid with subscription toggles. MSW: `GET /api/tags/summary`.
3. **`SearchResultsPage`** — Faceted results (Articles / Authors / Tags). Syncs with `?q=` query param. MSW: `GET /api/search?q=...&category=...`.

### Phase 5: Personalization & Notifications (MSW Mocked)

1. **`NotificationsPage`** — List grouped by date. Mark individual / all as read. MSW: `GET /api/notifications`, `PUT /api/notifications/:id/read`, `POST /api/notifications/read-all`.
2. **`CommunitiesPage`** — Followed communities list with toggle Join/Leave. MSW: `GET /api/communities/followed`, `POST /api/communities/:id/follow`, `DELETE /api/communities/:id/follow`.
3. **`DiscoverCommunitiesPage`** — Recommended communities gallery. MSW: `GET /api/communities/recommended`.
4. **`CustomFeedsPage`** — Feed builder with tag/community/sort-order criteria checkboxes. MSW: `GET /api/feeds/custom`, `POST /api/feeds/custom`.
5. **`ProfileSettingsPage`** — Display name, bio, avatar upload (presigned S3 URL flow). MSW: `GET /api/users/profile-settings`, `PUT /api/users/profile-settings`, `POST /api/users/avatar-upload-url`.
6. **`AccountSettingsPage`** — Password change with validation, account deletion with confirmation modal. MSW: `PUT /api/users/security/password`, `DELETE /api/users/account`.

### Phase 6: Analytics & Admin/Moderation Screens (MSW Mocked)

1. **`UserAnalyticsPage`** — Engagement metrics with 7/30-day filter. Custom SVG charts. MSW: `GET /api/users/:username/analytics/summary`.
2. **`DetailedAnalyticsPage`** (admin) — Platform-wide growth, demographic stats, CSV export trigger. MSW: `GET /api/analytics/community-detailed`.
3. **`AdminDashboardPage`** — Platform health metrics cards + status indicators. MSW: `GET /api/admin/metrics`.
4. **`UserManagementPage`** — Searchable user table; role and status change actions. MSW: `GET /api/admin/users`, `PUT /api/admin/users/:id/role`, `PUT /api/admin/users/:id/status`.
5. **`ModerationQueuePage`** — Flagged content queue; Approve/Hide/Warn actions. MSW: `GET /api/moderation/queue`, `POST /api/moderation/resolve`.

### Phase 7: MSW Handler Completeness & Missing API Documentation

1. **Finalize `handlers.js`**: All handlers include:
   - `delay(500)` for realistic latency simulation
   - Edge-case response variants (empty list, 404, 500) accessible via query flag (e.g., `?msw_scenario=empty`)
2. **Generate `src/docs/missing-apis.md`**: Full contract documentation for all 14+ missing endpoints — request headers, request body schemas, success and error response schemas, entity data types.
